**What the agent must build:**
A Flask customer-support API source tree with route decorators, JWT middleware, and
intentionally inconsistent authorization behavior is provided alongside extracted route
metadata and a large NDJSON access-log corpus. The agent writes Rego policy modules — it
does NOT edit the Flask API. The final Rego library returns normalized findings and a
Graphviz DOT attack-path graph that match golden output files.

**Milestones:**
1. **Map Flask Route Policies** — parse the Flask source and route metadata, produce a
   Rego `data.json` input document mapping each route to its declared decorator set,
   required JWT claims, and middleware chain. Output: `/app/route_policies.json`.
2. **Detect JWT Misuse** — write Rego rules over the NDJSON access-log corpus that emit
   normalized findings for each bypass: decorator/middleware mismatch, cryptographically
   valid JWT accepted without scope check, missing decorator on sensitive route.
   Output: `/app/findings.json`.
3. **Render Report Graph** — from the findings, generate a Graphviz DOT attack-path graph
   with canonical sorted node/edge order that matches the golden DOT file.
   Output: `/app/attack_graph.dot`.

## 2. Target difficulty + why it will hold

- **Target:** MEDIUM–HARD
- **Oracle gate (Gate 1):** LOW risk. Rego is our domain. No service to run — agent writes
  policy files, oracle runs `opa eval`. Fully deterministic.
- **Nop gate (Gate 2):** LOW risk. Without Rego files, `opa eval` returns nothing; tests
  compare against golden output which will differ.
- **Sufficiency gate (Gate 3):** LOW-MEDIUM risk. Document the three finding types and
  the mismatch rules precisely as requirements. Don't describe how to write the Rego.
- **Static gate (Gate 4):** LOW risk. Offline, OPA + Python baked in Dockerfile, golden
  files generated by oracle.
- **Difficulty (Gate 5):** MEDIUM-HARD. See §3.

**Why difficulty survives full documentation:**
The agent must correctly implement Rego rules that simultaneously check decorator
declarations, middleware runtime behavior, and JWT claim semantics — three interacting
sources of truth that each alone is insufficient. OPA's no-recursion constraint means
cross-route reachability analysis (which routes are reachable from a bypass entry point)
cannot use recursive rules and must be manually unrolled into a bounded fixpoint chain.
This is derivation work no amount of documentation removes.

## 3. THE difficulty lever

### Primary lever: correctness-density across three interacting rule axes

Each finding type requires agreement across all three axes simultaneously:

**Axis 1 — Decorator declaration** (from Flask source):
- Route has `@require_auth` decorator → claims `sub` + `role` required
- Route has `@require_admin` decorator → claims `sub` + `role=admin` required
- Route has no decorator → public (no claims required)

**Axis 2 — Middleware runtime behavior** (from route metadata JSON):
- Some routes marked `decorator: require_auth` in source but `middleware_enforced: false`
  in metadata (middleware bypass — the decorator is declared but not enforced at runtime)
- Some routes marked `decorator: none` in source but `middleware_enforced: true` in
  metadata (shadow enforcement — enforced without declaration)

**Axis 3 — Observed JWT behavior** (from NDJSON access logs):
- `jwt_valid: true` means signature verified, but does NOT mean claims were checked
- `scope_checked: false` means the route accepted a valid JWT without verifying `role`
- `status: 200` on a `require_admin` route with `role: user` = bypass finding

**The three finding types (all three axes must agree for a finding to be emitted):**
1. `DECORATOR_MIDDLEWARE_MISMATCH` — decorator says X, metadata says opposite
2. `SCOPE_NOT_CHECKED` — JWT valid, route requires role claim, but `scope_checked: false`
   observed in logs for that route
3. `UNENFORCED_SENSITIVE_ROUTE` — route has no decorator AND `middleware_enforced: false`
   AND route path matches `/admin/*` or `/internal/*` pattern

**Wrong-shortcut traps (one test per trap):**
- Trap 1: checking decorator presence alone without verifying `middleware_enforced` in
  metadata → misses DECORATOR_MIDDLEWARE_MISMATCH findings
- Trap 2: treating `jwt_valid: true` as equivalent to `scope_checked: true` → misses
  SCOPE_NOT_CHECKED findings
- Trap 3: flagging all public routes as UNENFORCED_SENSITIVE_ROUTE regardless of path
  pattern → produces false positives that golden file does not contain
- Trap 4: symmetric DOT edges (A→B and B→A) instead of directed (A→B only) → DOT file
  won't match golden output

**Secondary lever: OPA no-recursion constraint on reachability**
The DOT graph in milestone 3 requires computing which bypass entry points can reach
sensitive admin routes through the API's call graph. OPA forbids recursive rules, so
the agent cannot write `reachable[r] { reachable[s]; calls[s, r] }`. It must manually
unroll the reachability into N bounded relaxation steps (our proven HARD lever from the
Rego escape room task).

### Determinism controls
- OPA version pinned by sha256 in Dockerfile
- Findings JSON sorted by `[route, finding_type]` before writing
- DOT nodes sorted alphabetically; edges sorted by `[src, dst]`
- NDJSON corpus has fixed content (not generated at runtime)
- All Rego `opa eval` calls use `--format pretty` with sorted keys

## 6. Build order — follow this exactly, do not skip ahead

1. **Design the data/contract first** (before any code):
   - Decide the exact Flask API routes, decorators, middleware_enforced values, and which
     routes have intentional inconsistencies
   - Decide the NDJSON log format: `{timestamp, route, method, bearer_token, jwt_valid,
     scope_checked, claims: {sub, role}, status}`
   - Decide the findings JSON schema: `[{route, finding_type, evidence: {}}]` sorted by
     `[route, finding_type]`
   - Decide the DOT format: digraph, nodes sorted A-Z, edges sorted by src then dst
   - Write down the exact expected findings and DOT content for the fixture data — this
     becomes the ground truth for tests

2. **Set up `environment/Dockerfile`** — base image, OPA binary + sha256, tmux, asciinema,
   pytest + pytest-json-ctrf, Flask source + metadata + NDJSON corpus all copied in.

3. **Build oracle** — `solve1.sh`, `solve2.sh`, `solve3.sh` each as incremental deltas.
   The oracle Rego files + scripts produce the golden output files. `solve.sh` wrappers
   just call `solveN.sh`.

4. **Write tests** — `test_m1.py`, `test_m2.py`, `test_m3.py`. Each:
   - Has `class TestMilestoneN`
   - Recomputes expected values from source data (not hardcoded literals)
   - Has docstring on every test function
   - Tests the three wrong-shortcut traps from §3
   - `test.sh`: PWD guard → `mkdir -p /logs/verifier` → pytest → reward block

5. **Write `instruction.md` files** — plain prose, 150–200 words per milestone, no
   markdown structure (no headers, no bullets, no bold, no tables). Milestone 1 includes
   overall task context. No `/tests/` or `/solution/` references. No how-to hints.

6. **Fill `task.toml`** — see §7 below.

7. **Verify in Docker**:
   ```bash
   docker build --platform linux/amd64 -t jwt-bypass ./environment
   # Oracle: all three milestones → reward 1.0
   # Nop: reward 0.0
   # Offline: --network none oracle → reward 1.0
   ```

8. **Run agents** — 2–3 runs each of GPT-5.5 and Claude Opus 4.8. Target < 80% pass rate.

9. **Submit, generate rubric, edit rubric** (see §8).

- **Difficulty = derivation/correctness-density, not recall or footgun** (`00`, `02`).
- **All test deps in Dockerfile**; `test.sh` = PWD guard + mkdir + pytest + reward block only.
- **Document WHAT, never HOW** — no Rego syntax hints in instructions (`03`).

  hardcode expected values as literals in test assertions.
- **solve.sh = wrapper only**; implementation goes in `solveN.sh`.
- **Each solveN.sh is an incremental delta** — don't copy unchanged files across milestones.
- **Rubric checkbox unchecked** before sending to reviewer.
- **Difficulty is a noisy sample** — don't over-react to one reading; tune by adding an
  interacting rule, not rebuilding infra.
