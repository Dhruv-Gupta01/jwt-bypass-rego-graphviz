The findings from milestone 2 identify bypass entry points. Now write Rego rules in a file named attack_graph.rego using `package attack_graph` — OPA v0.68.0 is at /usr/local/bin/opa. The rules need to define sets named `nodes` and `edges` that compute which routes are reachable from those entry points through the API's internal call graph, and a partial object rule named `exposure` so that `data.attack_graph.exposure` returns a map of node to exposure count when queried via OPA eval.

The call graph is at /app/route_calls.json — keys are route paths, values are arrays of routes each one calls internally. A route belongs in the attack graph if it has a finding or is reachable within three hops from any finding route through that call graph. An edge src -> dst appears when both endpoints are in the graph and src calls dst. Reachability is bounded to three hops from any finding route. Each element of the edges set is a two-element array [src, dst]. The exposure count for a node is the number of distinct finding routes that have a directed path of at most three hops ending at that node (a finding route counts as its own zero-hop path to itself).

When loading data, wrap findings.json under key `findings` and route_calls.json under key `route_calls` into a single combined JSON object and pass it via `--data`. Write the output to /app/attack_graph.dot as a digraph named attack_paths. Node declarations come before edges. Nodes are sorted alphabetically, edges sorted by source then destination. Use quoted node IDs. Each node declaration uses the format: "route" [label="route" exposure=N] where the label equals the node ID.
